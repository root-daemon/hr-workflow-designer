Section 01

Executive Summary

A working prototype of an HR Workflow Designer, delivered within the 4–6 hour time-box. The application satisfies every mandatory requirement and implements four of the six bonus features.

🎨
Visual Canvas
Drag-and-drop React Flow canvas with 5 typed node varieties, live edge connections, and a minimap.
⚙️
Node Config Forms
Per-node dynamic forms with controlled components, validation, and type-safe state propagation via Zustand.
🔌
Mock API Layer
Pure async functions simulating GET /automations and POST /simulate with realistic latency.
⚗️
Simulation Sandbox
Step-by-step animated execution log, graph validation, topological sort, and raw JSON inspector.
↩️

📦
Import / Export
Full workflow graph serialisation to JSON — downloadable file or clipboard copy — and importable back in.

Section 04

State Management

All mutable application state lives in a single Zustand store. Components subscribe with fine-grained selectors, so a form edit never re-renders the canvas and vice-versa.

stateDiagram-v2 [*] --> Idle : app mounts Idle --> NodeAdded : addNode(kind, position) Idle --> NodeDeleted : deleteNode(id) Idle --> EdgeConnected : onConnect(connection) Idle --> ConfigEdited : updateNodeConfig(id, patch) Idle --> NodeSelected : selectNode(id) Idle --> Undone : undo() Idle --> Redone : redo() NodeAdded --> Idle : pushHistory → append node NodeDeleted --> Idle : pushHistory → filter node + edges EdgeConnected --> Idle : pushHistory → addEdge ConfigEdited --> Idle : patch node.data.config NodeSelected --> Idle : set selectedNodeId Undone --> Idle : restore history[index-1] Redone --> Idle : restore history[index+1]

Figure 3 — Zustand store state machine. Every mutating action optionally snapshots history first.

Store Shape

interface WorkflowState {
  // Core data
  nodes:          WorkflowNode[]
  edges:          WorkflowEdge[]
  selectedNodeId: string | null

  // History (undo/redo)
  history:        { nodes: WorkflowNode[]; edges: WorkflowEdge[] }[]
  historyIndex:   number           // pointer into history ring

  // React Flow handlers
  onNodesChange:  (changes: NodeChange[]) => void
  onEdgesChange:  (changes: EdgeChange[]) => void
  onConnect:      (connection: Connection) => void

  // Domain actions
  addNode:        (kind: NodeKind, position: XYPosition) => void
  deleteNode:     (id: string) => void
  deleteEdge:     (id: string) => void
  updateNodeConfig: (id: string, patch: Partial<NodeConfig>) => void
  selectNode:     (id: string | null) => void
  pushHistory:    () => void
  undo:           () => void
  redo:           () => void

  // Serialisation
  exportWorkflow: () => string        // → JSON
  importWorkflow: (json: string) => void
}
✓

Undo / Redo implementation: pushHistory() deep-clones the current { nodes, edges } before every mutating action. The ring keeps the last 50 snapshots. undo() decrements the pointer and restores; redo() increments it. No external library needed.

Section 08

Simulation Engine

The simulation runs in three phases: validate the graph structure, sort nodes into execution order via a topological BFS (Kahn's algorithm), then generate a SimulateStep for each node.

flowchart TD A(["/simulate called"]) --> B B{"≥ 1 Start node?"} B -- No --> ERR1["error: missing Start node"] B -- Yes --> C C{"≤ 1 Start node?"} C -- No --> ERR2["error: multiple Start nodes"] C -- Yes --> D D{"≥ 1 End node?"} D -- No --> ERR3["error: missing End node"] D -- Yes --> E E{"Any disconnected\nnodes?"} E -- Yes --> ERR4["error: list disconnected nodes"] E -- No --> F ERR1 & ERR2 & ERR3 & ERR4 --> FAIL(["return\n{ success: false, errors }"]) F["Build adjacency list\n+ in-degree map"] --> G G["Queue ← nodes with in-degree 0"] --> H H{"Queue empty?"} H -- No --> I["Dequeue node\nAppend to sorted[]"] I --> J["Decrement in-degree\nof all neighbors"] J --> K{"Any neighbor\nreaches in-degree 0?"} K -- Yes --> L["Enqueue neighbor"] --> H K -- No --> H H -- Yes --> M M{"sorted.length\n== nodes.length?"} M -- No --> N["Cycle detected\nFallback: original order"] M -- Yes --> O["Use sorted order"] N & O --> P P["For each node:\n• build message string\n• assign random durationMs\n• weighted-random status"] --> Q Q["Sum totalDurationMs"] --> R(["return SimulateResult"]) style FAIL fill:#4c0519,stroke:#f43f5e,color:#fda4af style R fill:#14532d,stroke:#22c55e,color:#86efac

Figure 8 — Simulation engine flowchart. Validation runs before sort; cycle detection falls back gracefully.

Kahn's Algorithm — Implementation

function topologicalSort(nodes, edges) {
  // Build adjacency list and in-degree map
  const adj      = {}   // nodeId → neighbor ids[]
  const inDegree = {}   // nodeId → number of incoming edges

  nodes.forEach(n => { adj[n.id] = []; inDegree[n.id] = 0 })
  edges.forEach(e => { adj[e.source].push(e.target); inDegree[e.target]++ })

  // Start BFS from nodes with no incoming edges
  const queue  = nodes.filter(n => inDegree[n.id] === 0)
  const sorted = []

  while (queue.length > 0) {
    const node = queue.shift()
    sorted.push(node)
    for (const neighborId of adj[node.id]) {
      inDegree[neighborId]--
      if (inDegree[neighborId] === 0)
        queue.push(nodes.find(n => n.id === neighborId))
    }
  }

  // If a cycle exists, sorted.length < nodes.length → return original order
  return sorted.length === nodes.length ? sorted : nodes
}
Section 09

Key Design Decisions

Each major architectural choice had at least one credible alternative. Here's what was chosen, what was rejected, and why.

State management library
CHOSENZustand — single flat store, actions co-located, fine-grained selectors, zero boilerplate, built-in snapshot for undo/redo
REJECTEDRedux Toolkit — powerful but excessive ceremony for a single-page workflow tool; adds 3× the boilerplate without meaningful benefit at this scale
REJECTEDReact Context + useReducer — viable, but causes entire subtrees to re-render on any state change unless carefully memoized; Zustand handles this transparently
API mocking strategy
CHOSENPure async functions in mockApi.ts — zero dependencies, trivially replaceable with real fetch, deterministic, no service worker setup friction
REJECTEDMSW (Mock Service Worker) — excellent for integration testing but adds a service worker registration step and complexity that isn't warranted here
REJECTEDjson-server — requires a separate process, breaking the "single npm run dev" DX requirement
Node form architecture
CHOSENSeparate form component per node type (open/closed principle) — NodeFormPanel dispatches on kind; adding a new type adds one file, changes one switch case
REJECTEDSingle generic form driven by a schema object — fewer files but harder to maintain as field types diverge; toggle vs select vs kv-list don't share a natural base component
Node style / colour system
CHOSENCentral nodeStyles.ts registry mapping each NodeKind to gradient / border / icon / badge classes — palette, canvas nodes, and forms all import from one place
REJECTEDInline styles per component — duplicates colour decisions, breaks consistency when a colour changes
Undo / Redo mechanism
CHOSENDeep-clone snapshot ring inside Zustand (50 entries) — straightforward, testable, no dependency on external libraries, works for any state shape
REJECTEDuse-undoable / immer-with-patches — more memory-efficient (patches vs full snapshots) but adds complexity; 50 snapshots of a typical workflow graph is ≈ 200 kB max
Styling system
CHOSENTailwind CSS v3 with a custom colour palette defined in tailwind.config.js — semantic tokens (bg-canvas, text-muted) keep components readable
REJECTEDCSS Modules — valid choice but requires context-switching between JS and CSS files; Tailwind keeps everything co-located

Performance Considerations

All node components are wrapped in React.memo. React Flow only re-renders nodes whose position or data reference changes. The Zustand selector pattern (s => s.selectedNodeId) means the form panel re-renders in isolation from the canvas on selection changes.

Type Safety

Every node's config is typed as a discriminated union via NodeKind. defaultConfig(kind) uses a satisfies assertion, so the TS compiler verifies each branch returns the correct shape. Adding a field to TaskConfig immediately surfaces in the form.
